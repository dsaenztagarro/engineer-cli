ASSISTED CAPTURE · engineer-cli · the draft-triage inbox & the git-source connect flow
Triage clears the way it should — open, glance, ⏎ to accept, x to reject, next. The git-activity source is native here: you commit from the terminal all day, so the drafts those commits become are best cleared where you already are. A draft is one proposed segment, never one-per-commit — the automation clusters a day's commits on a repo into a single ~50m build you accept, edit, or reject. Accepting writes an activity; the log stops lying without you reconstructing your week.
This canvas reads top to bottom: the pending inbox → open one draft → accept & reject in place → the ambient count → the git-source connect flow → the headless twins → the decision record.

FIRE-THEN-RE-READ · the write leaves the client's hands
accept ⏎ · what actually happens
⏎ accept
│
├─▶ PATCH :complete { resolution, response }
│ └─ fires on_complete → activity minted

├─▶ draft → completed (terminal)
│
└─▶ re-read pending — row gone; count drops
Same shape as Progress's “adjust may mint a successor.” The client owns no second ledger — the server's scope is the truth, and the CLI just re-reads it.

DECISION TO RAISE · where does “watching” live? click a model
⚠ Open contract — the one real cross-repo dependency
The shipped namespace :automations exposes task consumption only — no route registers a source. Tasks are “created in-process by AutomationJob.” So this flow is drawn against an unsettled contract. Two models:
server-side · local config · undecided
A · a new server endpoint
POST /api/v1/automations/sources
The engineer team adds a route; the CLI just calls it and the detector reads server state. Clean, but net-new server work this module can't ship alone.
B · purely-local client config
~/.config/engineer/sources.toml
Connect writes a local file the server-side git detector reads. No new route — but the detector must reach into client state, which may not fit the pipeline.
Raised, not resolved
The client UX above holds under either model — the repo picker and identity check are the same. What differs is only what ⏎ writes to. This is a design-pass flag for the handoff, not a decision the CLI makes unilaterally. (Click a model to see its shape.)

Connect flow · open contractThe shipped namespace :automations exposes task consumption only — no route registers a git source. Job 2's flow is designed against an unspecified contract: a new server endpoint (the engineer team adds it) or purely-local client config the detector reads. This is the one real cross-repo dependency — surfaced for the handoff, not invented.
Accept is a server writecomplete runs on_complete and leaves the draft in a terminal completed status; a second attempt is the 422 “Cannot complete.” So the CLI treats accept as fire-then-re-read — refresh the pending scope, never trust a cached row, and surface the 422 as “this draft moved on,” not a hard error. Same shape as Progress's “adjust may mint a successor.”
One segment, not one-per-commitThe automation clusters a day's commits on a repo into a single proposed segment — accepting writes one activity, not one per commit. The inbox exists to make the log honest, not to flood it. The CLI only ever consumes tasks: no create, no destroy — index / show + acknowledge / accept / reject + pending.
