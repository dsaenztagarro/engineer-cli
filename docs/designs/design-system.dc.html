DESIGN NOTES · BEHAVIOUR + WEB→TUI
◆ ACTIVITY FORM — what changed
Deduped the mode line. The form carried an in-panel -- INSERT -- row and a footer below the Block — the same state twice. ratatui draws the form Block, then one footer Line (mode badge + keys). Never two.
Trimmed to editable metadata. Books · repos · notes · segments left the create/edit form — they already live on the Activity detail. The form no longer mirrors the whole record; it fits ~24 rows instead of 44. :activity new and e on a row open this same short form.
◆ RATATUI behaviour (form)
• Each field = its own one-row Block. Focused field: border = accent 105, cursor = inverse cell.
• status · kind · domain · subdomain · bloom · path are closed pickers — ⏎/Space opens a popup List (Telescope-flavoured) over the form; never free text.
• tags = inline chips + input: ⏎ commits, ⌫ on empty removes the last.
• description = e → $EDITOR; the box only previews. No TextArea in the grid.
• :w writes (POST/PATCH), :q / Esc×2 cancels. Headless twin: engineer activity new --title … --kind ….
◆ WEB → TUI translation (review · coding-sessions · book-practice)
The web cockpits stack many rich panels on one page. In the grid, collapse each to List → detail, one job per screen:
• review — the due queue is a List; the drill is a full-screen single card + rate row (already shipped ✓).
• coding-sessions — session Table (repo · lang · intent · logged); detail = exercise ledger as a List, solved/unsolved via glyph + colour. Drop pill soup — intent is one styled Span.
• book-practice — the contents tree is a List with indent + ▸/▾ disclosure; one edition column, second on Tab. Exercise blocks (not individual rows) with a LineGauge; repos as a short list. No side-by-side edition columns.
Rule of thumb: one panel per screen, / to filter, ⏎ to descend, h to climb — depth replaces the web's width.

ERROR & NOTIFICATION MODEL · WHERE EACH MESSAGE LANDS
engineer-cli — every info / success / warn / error, one home each
Every message the client shows lands in exactly one of three places, chosen by scope × lifetime. The wording is identical across the TUI tile, the inline line, and headless stderr (§C) — one spelling per outcome. Colour is fixed: info accent · success 108 · warn 179 · error 167.
TIER 1FOOTER NOTIFY TILEtransient · auto-expiring · the outcome of a keystroke
Where: the bottom footer row — it borrows the key-hint line for a few seconds, then reverts. Never blocks input. Use for: the result of an action you just took, or an ambient state flip (online → offline).
ℹ search: type then Enter   info — accent text
✓ progress saved   success — green text
⚠ offline — showing cached books  warn — full-row bg
✖ log failed: request timed out  error — full-row bg
TIER 2INLINE PANEL STATEpersistent · scoped to one region · a read that failed
Where: inside the bordered block whose data didn't load — the body that would hold rows shows the reason + a retry key, while the rest of the screen stays live. Distinguish empty (no rows) from failed (couldn't fetch). Use for: a list / detail / segment load — one panel is broken, the screen isn't.

TIER 3BLOCKING SCREENpersistent · whole screen · nothing else can render
Where: the entire content area, with the one recovery action offered (retry / re-auth / quit). Use for: auth down or 5xx, expired token (401 → re-auth), missing config — the screen is meaningless without it. Reference: the SIGN IN · SERVER ERROR tile. At launch with no network, reads fall back to cached, read-only (§A) rather than blocking.
Pick the tier:  transient outcome of a keystroke? → Tier 1  ·  one region failed, rest fine? → Tier 2  ·  whole screen unusable? → Tier 3.
Honesty (§0·4): offline / stale / failed always render — never hidden to look tidy. One spelling (§C): the tile, the inline line and stderr share the same copy & a matching exit code, so what a user reads on screen is what a script greps.
