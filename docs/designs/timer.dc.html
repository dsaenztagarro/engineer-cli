Three verbs, one each: keep · trim (moves the idle span to paused time, timer runs on) · stop. Idle time is never logged as its own segment — no “work + idle” split.
j/k  choose  ·   ⏎  apply  ·   Esc  keep & resume
Presence in the CLI = keystrokes inside the TUI (a detached / closed session counts as absent). Idle is decided on the next interaction — this screen opens on launch, on g t, or on the first keypress after the threshold, mirroring the web’s next-interaction rule. No screenshots, no keylogging, ever.

Blocked on API — designed fully so the request is concrete (doc §C)
The v1 JSON API exposes the read (with idle / mode / phase / intervals_completed), start / pause / resume / stop / bind / candidates / discard, and segment CRUD. Web-only today, no API yet: the idle heartbeat & reclaim (§5), focus phase transitions & mode switch (§4), the segment audit list / acknowledge / delete-flow (§7), the overrun ping (§5), and settings (§7). These ship as blocked tickets raised against the engineer repo — the screens above spec each one so the ask is exact, not implied.
Out of scope here, tracked elsewhere: the weekly recap line, thin-week invite, the enriched Home leading with timer + pace, week-planning’s “start binds a planned item”, and the Progress pace / rollup screens the audit subtabs point at.
