§0 THE GOVERNING PRINCIPLE — sterling, not a replica the law every other brief is measured against · the glance-or-gesture test
An Apple Watch for the study loop, not a shrunk-down web app. The exemplar is already shipped — the timer: an ambient string you glance at, verbs that cost one keystroke, states that never lie, offers that never fire on their own, a headless twin you pipe into a status bar. That is the bar.
A full terminal replica of the web is a non-goal — explicitly. The web is the cockpit (editing depth, rich filtering, bulk ops, dashboards, the long tail of CRUD). The terminal owns the high-frequency core of the study loop, distilled. Port the web surface, or distil the one thing that matters here — the terminal always takes the second.
THE TEST — a feature earns its place only if it is
checked against every new feature, and every growth of an existing one, before it is designed
1
A glance or a gesture, not a session. Value lands in one look (a complication — a pace meter, the timer string, a due count) or one keystroke (a verb), never a lean-back editing workflow.
2
Ambient & quiet. Present without being opened; calm when on-track; a small signal when not; never a nag. On-pace is silence, behind is as loud as it gets.
3
Distilled, not ported. Answers the one question that matters in the terminal. Depth (the full table, the analytics grid, the planning canvas) stays on the web.
4
Honest. paused / idle / over / behind / stale / offline render truthfully — the design never hides a state to look tidy.
5
Composable — the terminal's edge over a watch. Every read is pipeable and every action is a headless verb (§C), so power moves to jq, git hooks and status bars instead of more on-screen chrome.
6
One-hand, keyboard-only, muscle memory. j/k, /, :, ⎵ leader, i/Esc — no mouse, no bespoke keymap.
A surface that fails the test belongs on the web.

THE TRAIN CASE · one write path, queued & reconciled
local action ──► optimistic state ──► sync queue │ (wire down) ▼ network returns ──► reconcile ──► merged / conflict surfaced
Acute for the timer, but the same queue-and-reconcile shape a target adjust, a log, or a note write rides — so it's briefed here, not smuggled into one module. When it lands it's the largest change to the client's I/O model: its own gap brief → epic.

§C THE TUI ↔ HEADLESS CONTRACT every read the TUI shows is also a one-shot · engineer timer is the reference — copy it
THE REFERENCE · src/timer_cli.rs — a machine form and a pipe form
bash — piped, NO_COLOR-respecting, TTY-detected
$ engineer timer --json { "state":"running", "elapsed_s":1453, "mode":"work", "activity_id":418, "kind":"build", "title":"Raft…" } $ engineer timer # stable, field-ordered plain line running 1453 work 418 build "Raft leader election" └─ state ─┘ └s┘ └md┘ └id┘ └knd┘ └───── title ─────┘ · absent field = - $ engineer timer --short # the status-bar reduction ● 24:13 # empty when nothing runs $ engineer timer --json | jq -r .elapsed_s 1453 # clean text: no ANSI when piped
Colour is gated on stdout().is_terminal() AND NO_COLOR being unset — a pipe or NO_COLOR=1 gets clean text. The plain line's column order never changes and uses - placeholders, so awk/cut scripts don't break.
EXIT CODES · they answer one question — is the clock counting?
0counting
1nothing running
3idle, reclaim pending
4not counting (paused / focus break)
Write verbs exit 0 on success, 1 on refusal, reason on stderr — and that refusal copy matches the notify-tile the screen shows for the same mistake. One spelling of every outcome.
Every NEW verb inherits this — no exceptions
engineer targetdeclare / adjust / retire + a list read — each --json + a piped row.
engineer logthe after-the-fact capture verb (owned by activities).
engineer pacethe headless twin of the pace meters — one line piped + one status-bar reduction, quiet by default.
If a module adds a read the TUI renders, it adds the one-shot in the same slice — the twin isn't a follow-up, it's half the feature. This is the definition of done those briefs cite.

§E $EDITOR FOR PROSE the git commit pattern · never rebuild a textarea inside ratatui
THE HANDOFF · spawn $EDITOR on a temp file, read it back on save
note body — press i → the TUI suspends, $EDITOR takes over
┌─ ratatui ──────────────────────────────────────────┐ │ title Raft consensus summary │ ← single-line: i/Esc │ body [ i ] open $EDITOR ▾ │ └────────────────────────────────────────────────────┘ │ i ▼ nvim /tmp/engineer-note-4f2a.md # suspend TUI, spawn │ :wq save # read the temp file back │ empty buffer # → cancels, no write ▼ ✓ body attached, TUI resumes
A TUI is a character grid with a one-line input idiom; a multi-line editor with the user's own keymaps, syntax and muscle memory already exists in $EDITOR. Reimplementing a worse one is the wrong medium. Save-vs-abort and the empty-buffer-cancels convention are owned in detail by notes & week-planning.
The line — body vs label
$EDITOR the body. Note bodies, retro reflections — anything past a single line. Spawn the editor, read it back.
i / Esc the label. A title, a query, an activity name — short single-line inputs stay in the app's insert grammar.
Stated here only as the shared principle
So no module invents an in-TUI textarea. The temp-file handoff, save-vs-abort semantics, and empty-buffer-cancels convention are specified where they're used — notes and week-planning.
