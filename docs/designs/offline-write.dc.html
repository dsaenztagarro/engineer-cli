diverged
Reconciliation needs a choice. Replay found the server moved on. This one is loud — danger, a full-row treatment — because the promise is: never silently drop or merge a segment. Surface it, let the user pick a side.
Quiet by default, loud only for a choice. The read cache's ` ~` and the local clock's ◔ already taught the eye "the number is real but unconfirmed." ↑N queued joins that family; diverged is the one moment we break quiet — on purpose.
A queue of intents is not a second ledger
Measure's rule is derived, never stored — the client keeps no second ledger. The queue is the deliberate, scoped exception: it stores pending intents, not derived actuals, and only until they sync. The server stays authoritative — the moment an intent lands, it leaves the queue.
One clock, offline too. The local clock uses engineer's study-day boundary and the same started_at / elapsed_seconds / paused_seconds / paused_at arithmetic the server does, so a locally-run session reconciles to the second. Not a second definition of time — the same one, computed client-side.

THE SEAM · every write already funnels through one place — the queue keys off it
local action (start / pause / stop / log …) │ ▼ api/mod.rs · send() │ ┌────┴─────────────────┐ │ │ 2xx ApiError::Transport │ │ (wire down) ▼ ▼ confirmed ──► enqueue intent ─► queue.json (XDG state) │ network returns ▼ replay in order ─► reconcile │ ┌───────────┴───────────┐ ▼ ▼ merged diverged → ask
The read cache already falls back to disk on Transport alone (auth / other errors still propagate). The queue keys defer on the identical seam — a blind byte-interceptor at send() won't do: each write returns a different typed resource the caller consumes, so the queue synthesises a plausible local response per verb, then reconciles it.

EXIT CODES · scripts branch on the code
engineer queue · exit ∈ { 0 · 3 · 4 · 5 }
0drained · nothing pending, nothing diverged
3queued · writes pending, offline (not an error)
4diverged · a choice is waiting
5replay failed · non-transport error, propagated
queued is deliberately not a failure code — offline is a normal state, so a cron job doesn't page on it. Only 4/5 want a human.
BLOCKED ON APIThe contract replay-and-reconcile needs — recorded, not faked client-side
1 · Idempotency under replay
If a queued POST /timer or create_segment is re-sent after the server already recorded it (ack lost on a flaky link), does the server dedupe on an idempotency key, or double-write? Without the guarantee, replay can duplicate segments — the opposite of "never lose one." Divergence case D above degrades to a prompt precisely because this isn't confirmed.
2 · A structured conflict the client can render
When the server has moved on, does it return a machine-readable conflict — like targets' 422 ClosedVersionError — that the reconcile screen turns into a choice, or a generic error? Cases A–C are designed to a structured conflict; a generic 500 collapses them into "something went wrong," which fails the honesty test.
Per the /epic backend-gap rule, these are blocked tickets, not client-side fakes: the queue mechanics (Foundation + Phase A–C enqueue/replay) can land ahead of the contract, but honest divergence resolution (job 4) is gated on it.  ·  Accent note: this canvas is drawn to 105 periwinkle per CLAUDE.md; the 105-vs-75 ratification is flagged in cross-cutting §D, not silently resolved here.
