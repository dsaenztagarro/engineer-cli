## [Unreleased]

### Changed

- **The accent is now indigo, not sky-blue — the brand hue, ratified.** `theme.rs` moves `ACCENT` from `256 #75` (a sky-blue that had drifted off the web's indigo hue) to periwinkle `256 #105`, with the selection background following from steel `67` to indigo-dim `61`. ## [0.6.0] - 2026-07-08

### Added

- **`engineer today` — the Home screen, headless.** A one-shot read of the composed daily-loop aggregate (`GET /api/v1/today`): `--json` emits the payload (date · timer · pace · plan · totals · review · reading), and the bare form prints a compact, greppable summary reading top-to-bottom like the screen. Read-only (Home owns no write); exits `0`, with `401`/transport errors surfacing through the shared `ApiError` path, matching the `engineer timer`/`target` contract. (`home.dc.html` §ONE READ)

- **One `g`-goto grammar on every screen.** Navigation gains a vim-style `g` prefix — `g t` timer, `g p` progress, `g r` review, plus `g h` home, `g b` books, `g n` notes, `g a` activities — so the same goto chord works everywhere and the footer advertises it (a `g`-menu appears the moment you press `g`). The existing single-letter jumps (`t`/`p`/`R`/…) and the `Space` leader stay as aliases, so muscle memory is intact. To make `g` a clean prefix, a list's "jump to top" is now **`gg`** (matching the documented `gg`/`G` grammar and vim); `G` still jumps to the bottom. Frees the single letters for screen-local actions going forward. (`home.dc.html` §Home · enriched footer)

### Changed

- **Home opens the day with the timer, from one read.** The Home screen is rebuilt to lead with the running timer as a band — the same `timer_cell` grammar the header speaks, larger — over today's plan (state glyphs, kind, logged-vs-planned, carried-over rows), a logged/review/left strip, and the books you're mid-chapter in with where-you-are. It all comes from a **single `GET /api/v1/today`** aggregate: the old two-call load (`list_activities` + `list_books`) and its client-side day-window math are gone, so Home now honours the server's 4 AM study-day boundary and can't drift from the header cell or Progress. Empty plan and idle timer render as calm states, never blank panels. (`home.dc.html` §Home · enriched / §nothing planned / §nothing running)

- **…and the pace fold leads beneath it.** The lead band gains this week's promise, pre-folded to the single worst-behind target: `behind` names the target by scope with a warn ` behind Xh ` chip and how many trail (`g p → Progress` for the full meters), while nothing-behind renders the calm `✓ This week · on pace` line — silence is the on-pace state, never a red or empty panel. The deep now-tick meters stay on the Progress screen (the `/today` fold carries only the delta, not the fill/target). (`home.dc.html` §Home · behind / §on pace / §THE LEAD)

## [0.5.0] - 2026-07-07

### Added

- **`engineer target` — declare, adjust, retire, and list weekly targets, headless and in place.** The Progress pillar gains its write side. A new targets client (`src/api/targets.rs`) over the shipped `/api/v1/targets` routes powers a headless verb suite — `engineer target list` (one row per lineage, `--all` / `--retired`), `declare` (exactly one scope of `--domain` / `--kind` / `--intent`, plus `--hours`), `adjust <id> <hours>`, and `retire <id>` — with `--json` (an array for list, an object per write), a stable plain-pipe form, TTY- and `NO_COLOR`-aware colour, and exit `0`/`1` for success/refusal, matching the `engineer timer` contract. On the **Progress screen** a target-row cursor (`j`/`k`) gains `e` to adjust the selected target's weekly hours in place and `x` to retire it (confirmed on a second press, history kept); the empty state now teaches `engineer target declare` instead of pointing at the web. It honours the server's append-only lineage (engineer ADR 0026): `adjust` returns the live row even when the edit minted a successor id, `retire` never deletes, and a closed-version `422` surfaces as "this target moved on — re-fetch" rather than a hard error. Interactive *declare* (which needs a domain/kind/intent scope picker) and a `:target` palette verb (it collides with the shipped `:t` → timer prefix) are deliberately deferred and recorded in the progress and command-palette briefs.

- **Presence heartbeat — the idle guard no longer trips on real in-TUI work.** While a timer is running (and not paused), a key in the TUI now marks presence: a background `POST /api/v1/timer/heartbeat`, throttled to at most once a minute (matching the web pill). So a session spent reading, navigating, and typing *inside* the TUI without touching timer verbs keeps the clock alive instead of drifting into the idle guard. It's deliberately silent once the timer has already gone idle — the reclaim screen owns that decision (its **Keep** verb is the explicit "I was present"), so moving through the reclaim list never auto-resolves it. Needs the engineer API's new `POST /api/v1/timer/heartbeat` (dsaenztagarro/engineer#781), the last v1 gap versus the web timer — closing the follow-up left open when the idle guard shipped server-driven in #35.

### Changed

- **Design briefs reorganised — one brief per engineer-cli module, under a governing product principle.** The single omnibus `terminal-client.brief.md` — stale in both directions (the CLI had shipped most of its Phase-1 jobs; the `engineer` API had shipped `targets` / `today` / `weeks` / `automations` ahead of it) — is retired to a tombstone and decomposed into one brief per module: `timer` (absorbing the former `timer-gaps` gap analysis, its backend-API "Section C" now resolved), `progress`, `home`, `activities`, `review`, `notes`, `command-palette`, `week-planning`, `assisted-capture`, and a `cross-cutting` brief. Each is grounded in the API it actually consumes (verified against `engineer/config/routes.rb`) and states shipped reality rather than a pre-implementation snapshot; the pre-lifecycle `daily-loop.brief.md` is marked historical and the retired `books.html` / `terminal-tokens.css` anchors corrected to `design-system.dc.html` throughout.

- **A governing principle — "sterling, not a replica."** A new lead section in `cross-cutting.brief.md` (pointed to from the kit `README` and the briefs index) makes explicit that a full terminal replica of the `engineer` web UI is a non-goal: the terminal owns the study loop's high-frequency core as glances and gestures — the shipped timer is the exemplar — with depth staying on the web. Three surfaces were reframed to fit it: progress's time-explorer becomes a glance plus a `--json` rollup rather than a pivot grid, the activities table is capped as the one deliberate lean-back ledger (its watch-native core being capture and the audit), and week-planning is a planned-vs-done readout plus a one-liner declare rather than a planning canvas.

## [0.4.0] - 2026-07-06

### Added

- **THIS WEEK on the instrument rail.** The watch face's rail completes its third instrument: a mon→sun sparkline over the progress read's new per-day series (`by_day`), today's bar in accent, scaled to the week's best day — above the TODAY total it complements. On servers without the series the rail quietly degrades to the today-only block. (`timer.dc.html` §Timer hero, the piece #29 shipped without)

- **The segment audit — find & fix the rows that don't look right.** `Progress ▸ a` (or `:audit`) opens the flagged-segments screen: rows grouped by what's wrong — *implausibly long* (over your long fence), *zero/near-zero*, *missing metadata* — with the flag words inline and the flagged total in the panel title. Row actions per the design's verbs: `a` **looks right** acknowledges the segment (the duration-shape flags clear permanently and a fully-clean row leaves the list; missing-metadata flags survive until an actual fix), `t` **trim** is a segment-edit preset — one PATCH that shortens the duration to your long fence, the value that makes the row plausible (guarded: only offered on long rows), `d` **delete** asks twice on the same row, and `f` **fix** hands off to the Activities table naming the activity. A clean log renders "no flags, no badge, anywhere". (`timer.dc.html` §Segment audit)

- **The overrun ping — amber, honest, never auto-stopping.** When cumulative time on the bound activity (earlier segments + the live clock) crosses its planned duration, the number turns amber: the watch face reads `● PAST THE PLAN — STILL COUNTING` with `planned 2h · logged 2h 18m` and the reminder that it never stops anything for you; the header cell on every screen wears the amber ` over ` pill; and a once-per-timer warning tile pings when a poll first crosses the plan (the server gates the whole thing on your `overrun ping` knob — off means off). Headless: the status word `over` (exit stays 0 — it's still work) with `planned_s=` in the contract line, an amber `--short` glyph, and `"over"`/`"planned_minutes"`/`"logged_minutes"` in `--json`. A focus break outranks the alarm — a break isn't counting. (`timer.dc.html` §Overrun)

- **Focus offers — nothing fires on its own.** The focus rhythm's transitions are now real and always yours to accept: when a work interval crosses your configured length the face reads `◆ INTERVAL n COMPLETE` and offers `b start 10m break · n skip — arm interval n+1` (every Nth break is the long one, with its own duration — both knobs from settings); when the break is done it reads `○ BREAK'S OVER` and `b` goes back to work. `n` mid-work banks the interval and immediately arms the next; `m` switches a running timer stopwatch ⇄ focus **in place** (elapsed preserved — entering focus opens a work phase). The start picker's `Tab → Focus` now starts a real focus session (start + mode hop) and quotes your durations. Away from the timer screen, a finished phase shows as an accent ` break? ` / ` work? ` pill on the header cell of every screen — `Space t` jumps in to decide. In stopwatch mode `b` keeps its bind meaning. (`timer.dc.html` §Focus offers)

- **The idle guard — welcome back, the clock went quiet.** When the server flags the timer idle, the timer screen opens the reclaim list: *what was the idle tail worth?* One row per server verb — **Trim** (idle becomes paused time, the timer keeps running — the safe pick), **Keep** (the tail counts), **Stop at last input** (saves the worked span, ends the timer, lands on the normal stop confirmation with undo) — plus the **Discard** escape, which rides the usual two-press confirm. The list preselects your `idle default reclaim` setting, `j/k` chooses, `⏎` applies, `Esc` defers (the guard returns while the read stays idle; the watch face meanwhile reads `◐ IDLE — RECLAIM PENDING`). Captions are computed from the read (`last_interacted_at`): running total, idle span, what each verb keeps. Nothing is written until you choose, and idle time is never logged as a segment. Headless: `engineer timer reclaim <trim|keep|stop>` and `engineer timer stop --reclaim=<verb>`; `timer status` now appends `idle_s=N` in the idle state. (`timer.dc.html` §Idle reclaim / §Overnight boundary)

- **Timer settings, readable everywhere.** The per-user timer knobs the web app edits are now first-class in the CLI: a read-only **Settings · Timer** screen (`:settings`) lists all twelve — default mode, the focus durations and long-break cadence, the idle guard and its threshold, the default reclaim action, the audit fences and badge, the overrun ping — with the explicit "edit on the web" pointer (one writer, no divergence). Headless twin: `engineer timer settings` (aligned table) and `--json` (mirrors the server payload 1:1). (`timer.dc.html` §Timer settings)

### Fixed

- **API layer aligned to the shipped v0.75.0 timer contracts.** The timer read now decodes the server's actual field names — `last_interacted_at` (the presence mark; previously modeled as `last_input_at` and silently `null`), plus `paused_at`/`paused_seconds`/`phase_started_at` and the overrun trio `planned_minutes`/`logged_minutes`/`over` — and the segments client uses the real nested routes (`/api/v1/activities/:activity_id/segments/:id`); the post-save undo passes the activity id accordingly.

## [0.3.0] - 2026-07-06

### Added

- **The focus session has a face.** When the live timer is in focus mode (today: started from the web; the CLI start joins once the focus API ships), the watch face reads it faithfully: `◆ WORK · INTERVAL n` with the accent digits while counting, and a muted `○ BREAK — NOT COUNTING` face with the reminder that a break is never a segment. The instrument rail gains the POMODORO block — banked intervals as green dots, the live one accent (hollow on a break), no invented remainder since the round length is a settings knob the API doesn't serve yet. `n` (skip interval) joins `m` in naming the missing focus API instead of silently doing nothing; the header cell's `◆`/`○ break` forms light up from the same read. (`timer.dc.html` §Focus rhythm)

- **Bind-at-stop, undo, and the discard that asks twice.** Ending an *unnamed* timer no longer dead-ends in a warning: `s` freezes the clock (a real pause, so deciding time never pollutes the segment) and opens the bind picker — `⏎` binds and saves in one stroke, `Esc` resumes exactly what the panel froze (a clock you'd paused yourself stays paused). The stop confirmation now carries **undo**: `u` deletes the just-written segment — the exact inverse of the save — and returns to the empty face. And `d` discard works on any live timer but respects real work: past ~2 minutes the first press warns (`discard 41m of work? d again to confirm`), only the very next `d` confirms, and any other key disarms it — the same fence `engineer timer discard --force` mirrors headless, now shared as one constant. (`timer.dc.html` §Bind at stop / §Saved & undo)

- **The start picker — one list, every way in.** `s` on the empty timer screen (or `/` from the live face) opens the §Start-a-timer picker: a live fuzzy search over your bindable activities, a `＋ new activity` row that creates & starts from the typed title, and a `▶ just start` row for an unnamed clock you bind when you stop. `Tab` toggles Stopwatch ⇄ Focus for the fresh start — picking Focus currently explains that focus-start needs the focus API rather than pretending. With a timer already running the picker becomes the stop-&-switch flow: the list narrows to existing activities, the first `⏎` shows the "One timer at a time" banner naming what's running and what you picked, the second `⏎` stops & saves the current timer and starts the new one (`Esc` steps back without touching the clock) — the same `switch` semantics as `engineer timer start --switch`. (`timer.dc.html` §Start a timer / §Start conflict)

- **The timer status line speaks every state.** The persistent header cell (every screen, right-aligned) now renders the full grammar from the timer redesign: `●` running in green with the muted activity title (truncated past 24 chars), italic *untitled* for an unbound clock, `‖` paused with the frozen clock muted, `◐` idle with an amber ` idle ` pill when the server flags a reclaim pending, `◆` focus work with pomodoro dots (banked green, live accent), and `○ break … not counting` muted. Below ~70 columns the cell drops to the narrow glyph-plus-clock form so the breadcrumb keeps its room — the fixed-footprint promise. This supersedes the v1 pill's "never a title" rule by design (`timer.dc.html` §Status line); the `over` form lands with the overrun feature.

- **The timer screen is a watch face.** The live timer renders as the redesign's hero (`timer.dc.html` §Timer hero / §Paused): one big block-digit number (accent while counting), a state label, the "since HH:MM" context, and the bound activity — or italic *untitled* with the bind hint. `i` folds the instrument rail (a TODAY total summed from today's activities; the focus instruments arrive with the focus display) and the number recenters into the calm face. `SPC` is pause ⇄ resume on the live face (`p` stays as an alias), and pausing freezes the whole face amber — `‖ PAUSED — NOT COUNTING`, muted digits, and the reminder that a paused timer never goes idle. `s` is the stage-dependent primary: start the clock when absent, end & save when bound, and the bind-first warning when unbound (`x` remains a hidden legacy alias). `m` explains that mode switch needs the focus API rather than silently doing nothing.

- **Headless `engineer timer` — every timer verb has a one-shot twin.** The live timer is now fully drivable from scripts and status bars, no TUI required: `engineer timer` prints a one-line human read (`--json` for the full read: state word, mode/phase, intervals, activity, timestamps, elapsed); `engineer timer status` emits the fixed-order contract line `<state> <elapsed_s> <mode> <activity_id> <kind> "<title>"` (with `-` placeholders when unbound) and `--short` the glyph-plus-clock status-bar form (`● 52:14`, empty when nothing runs) — the same glyphs as the in-app header cell. Write verbs: `start [query]` fuzzy-binds via `/timer/candidates` (bare `start` begins an unnamed timer; a conflict refuses and names what's running, `--switch` stops & saves it first), `toggle` (pause ⇄ resume, made for tmux/zellij keybinds), `pause`, `resume`, `stop` (refuses on an unbound timer with "bind or discard first"), `bind <query>`, and `discard` (past ~2 minutes it refuses without `--force`, the headless form of the TUI's confirm). Exit codes answer *is the clock counting?* — `0` counting (running/focus work) · `1` nothing running or a refused write · `3` idle, reclaim pending · `4` paused or on a break — and output is plain when piped (ANSI only on a TTY, `NO_COLOR` honoured). (`timer.dc.html` §Headless / §Headless contract)

- **Book status picker modal — the last of the daily-loop screens.** Pressing `s` on a book's detail opens a small centered modal listing the five statuses as their kit pills (` reading ` accent · ` done ` success · ` unread ` muted · ` hold ` warn · ` stop ` danger), opened at the book's current status. `j`/`k` step the highlight and the `r`/`c`/`u`/`h`/`a` mnemonics jump straight to a status; `↵` confirms and `Esc` cancels. Confirm PATCHes `/api/v1/books/{id}` via the existing `update_book`, the detail header reflects the new status, and a notify tile confirms (`status → on hold`). While the modal is open it owns those keys (so `h` picks `on_hold` rather than stepping back, and `Esc` closes the modal, not the screen). This finishes the sketched-but-unbuilt TODO on the book detail screen and completes EPIC #7's phased plan for the terminal daily loop.

- **The `:` command palette, completed as one grammar table.** The command line now knows the whole daily-loop vocabulary pinned by the brief (§5, `command-palette.html`), and the whole thing is driven by a single source of truth — `src/app/command.rs`'s `ENTRIES` table (verb → kind → argument shape → target + one-line help). The dispatcher, Tab completion, the inline hint, and `:help` all read that one table, so what runs can never drift from what the UI advertises (the old scattered `if`-chain in the submit handler is gone). **Navigation** covers every screen — `:home` `:books` `:activities` `:notes` `:review` `:progress` `:timer` (bare verb goes to the screen); **actions** are `:timer start|pause|resume|stop` and `:note <text>`; **housekeeping** is `:q`/`:quit` `:logs` `:w` `:logout`, plus a new `:help`. Timer actions dispatch against the app-owned timer snapshot from *any* screen (no navigation, no screen-load race) — the header cell shows the result, and an invalid transition surfaces the same notify-tile warning the Timer screen gives (e.g. stopping an unbound clock says "bind the timer before stopping"). `:note <text>` opens the quick-capture overlay **prefilled** with the text (a *new* draft, not a direct create) so a thought can grow an anchor or be saved with one keystroke. The line reads its four states from the table: an empty `:` advertises the top verbs, a partial verb shows the matches and completes on **Tab** to their longest common prefix (vim's `wildmode=longest`; stateless, no cycle to track), an unknown verb is helpful not hostile ("unknown :xyz — try :help"), and executing shows the normal result. **Abbreviations** resolve as in vim: an unambiguous prefix wins (`:act` → activities, `:t start` → timer start, `:timer p` → pause), while an ambiguous one lists its candidates instead of guessing (`:l` → logs/logout, `:h` → home/help). The unpinned `:activity`/`:a` verbs were dropped (the +activity form stays on the `a` key), which is what lets `:act` and `:a` resolve to activities.

- **Review screens — dashboard, the rating sitting, and browse (`R` from Home, the `<Space>` leader `R`, or `:review`).** The Review pillar's first UI: one screen, three stages. The **dashboard** is a minimal read — the due-queue count and estimated minutes as the call to action, the streak stats (`streak N days · best M · X this month · avg Yd interval`), and the due queue in urgency order as a preview. No ASCII heatmap: the streak and this-month counts already convey cadence in the terminal, and the web app owns the heatmap (the `dashboard.heatmap` payload is parsed but deliberately not rendered). The **sitting** is the payoff: `↵` (or `s`) starts at the queue head, each topic shows its name, domain, and last-reviewed/interval context (plus prompts and the per-rating interval forecast when the payload carries them), and you rate with a single keystroke — `f` forgot · `z` fuzzy · `s` solid · `i` instant (chosen so the ratings never collide with the global `r` refresh). The server returns the next due topic and the sitting advances automatically until the queue drains to a quiet "done" view (rated count); `Esc` exits mid-sitting cleanly, since each rating is already committed per topic. **Browse** is the secondary state (`b` from the dashboard): a paginated topic table where `s` cycles the API's sort ring (urgency → recent → most/least-reviewed → longest-interval → a–z), `/` maps to the server-side `q` search, `]`/`[` step pages (the Activities/Progress precedent), and `↵` opens a topic detail read with a one-off rate option. The API client (`src/api/review.rs`) was vendored byte-identical from the user's WIP tree (the #9/#10 discipline), diff-verified before and after the work.

- **Activities table screen (`A` from Home, or `:activities`).** The core domain surface: a dense, scannable ledger of recent activities — a semantic status pill, kind, title, domain (by name; the terminal palette has no per-domain colours), duration, and a relative "when" (`5m ago`, `2d ago`, then a date). It is the **first surface to expose `meta.page`**: the panel's bottom border reads `page N of M · X total`, and `]`/`[` step pages (consistent with the Progress screen's week nav). `f` cycles a single TUI-light filter ring (all → planned → started → completed → archived) that folds the server's status and archived axes into one key; `/` filters the loaded page client-side across title, kind, and domain (the list API has no text search). Row actions: `c` completes, `a` archives/unarchives (reversible, so it toggles quietly with no confirm), `d` duplicates ("do this again"), `↵` opens a full-field detail read (kind/intent/bloom, domain, duration, segment count, timing, generated notes), and `t` starts the live timer bound to the selected activity (the header cell refreshes). Mutations refetch the current page rather than patching a row, so the ledger always mirrors the server. `A` opens the table while `a` stays "+activity" (the new-activity form), keeping muscle memory intact. Home's collapse to a single `GET /api/v1/today` call is deferred until that endpoint ships (engineer#663).

- **Quick-capture overlay + notes browser.** Capturing a note now costs a couple of keystrokes from anywhere: `<Space>c` (or `c` on Home/Notes) opens a modal quick-capture overlay over the current screen — a multiline thought, an optional book anchor (a live search over your books, the timer bind-panel idiom), and an optional page. Save is explicit (`Ctrl-S`, since Enter is a newline in the editor); **capture is sacred** — an accidental `Esc` on a non-empty draft only warns (`Esc` again to discard), so input is never lost. On save the note POSTs to `/api/v1/notes`, adding a book + page anchor when both are set. A new **Notes browser** (`n` from Home or the `<Space>` leader, or `:notes`) lists notes with a one-line anchor read-back for anchored notes (`SICP · ch 3 · p.142`) next to loose thoughts, `/` searches the server, `j`/`k` navigates, `↵` opens a full-content detail read, `a` archives/unarchives, `t` reveals archived notes (dimmed in place), and `e` reopens the same overlay pre-filled to edit (PATCH).

- **Timer screen + persistent header cell.** A new `Timer` screen owns the daily-loop timekeeping moments: start a blank clock ("name it later"), bind a running timer to work by searching `/api/v1/timer/candidates` or minting a new activity from a typed title, pause/resume, and stop — which shows the written segment (minutes → activity) so the ledger is trusted. On top, a fixed-width **header cell on every screen** renders the live timer per the web pill contract: an accent `● mm:ss` (widening to `h:mm:ss`) when running, an amber `‖ mm:ss` when paused, and nothing when absent — never the activity title. The snapshot lives in app state and is polled every 15s from `GET /api/v1/timer`, with the displayed elapsed ticked locally between polls. Reachable from Home with `t` (or the `<Space>` leader `t`, or `:timer`); on-screen keys are `s` start · `/` bind · `p` pause/resume · `x` stop · `d` discard.

- **Progress screen (`engineer pace` meters).** A new read-only screen renders the weekly pace aggregate from `GET /api/v1/progress` as one ASCII meter per target, behind-first: a block-bar with a now-tick (where the week expects you to be), `actual/target` hours, the signed delta, and the pace state (`behind` amber / `on pace` green / `met`). A week-header line (`2026-W27 · sat · day N of 7 · now = NN%`), a behind-total footer naming the largest gap, and a compact kind-mix line round it out. Reachable from Home with `p` (or the `<Space>` leader `3`/`p`); step weeks with `[` / `]` and return to the current week with `t`. Pace is derived from segments at read time, so nothing is stored client-side.

## [0.2.0] - 2026-06-21

### Changed

- **Renamed the project from `engineer-tui` to `engineer-cli`.** The package name, OS keyring service, XDG config/state directories (`~/.config/engineer-cli/`, `~/.local/state/engineer-cli/`), log filename prefix, tracing namespace (`engineer_cli::api`), log env var (`ENGINEER_CLI_LOG`), and the CIMD `client_id` document path (`engineer-cli.json`) all moved to the new name. ### Added

- **Prebuilt binaries and Homebrew distribution via [cargo-dist](https://opensource.axo.dev/cargo-dist/).** Tagged releases now cross-build macOS (arm64/x86_64) and Linux (arm64/x86_64) binaries, publish them to GitHub Releases with a shell installer, and push an `engineer` formula to the [`dsaenztagarro/homebrew-tap`](https://github.com/dsaenztagarro/homebrew-tap) tap. Install with `brew install dsaenztagarro/tap/engineer`.

## [0.1.0] - 2026-06-19

### Added

- Initial release: a neovim-flavored terminal client for [Engineer](https://github.com/dsaenztagarro/engineer) to log activities and track reading progress, authenticating against [Identity](https://github.com/dsaenztagarro/identity) over OAuth2 Authorization Code + PKCE (RFC 8252).
- **Zero-friction authentication via an OAuth Client ID Metadata Document (CIMD).** The client identity is the URL of a metadata document served by the Engineer app; Identity fetches it server-side. The `client_id` is derived from `api_url`, so no manual client registration or pasted credentials are needed.
- **In-TUI sign-in screen.** Launching without a stored refresh token lands on a "Sign in" screen; pressing Enter runs the browser OAuth flow and drops into the authenticated TUI. The refresh token is stored in the OS keyring; subsequent launches go straight in. The `login`, `logout`, and `whoami` subcommands remain for scripting.
- **Explicit environment selection** via the `--env` flag and `ENGINEER_ENV` variable (`production` default, or `development` for localhost), with built-in URL presets so a fresh run needs no config file.
