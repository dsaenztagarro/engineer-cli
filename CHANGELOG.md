## [Unreleased]

### Added

- **The error/notification model across the detail & aggregate screens — BookDetail, Progress, Week, Audit.** Each one-read-one-aggregate screen now shows a loud Tier-2 `✖ couldn't load …` + `r` retry when its read fails, instead of a blank or half-drawn body — while the header timer, footer, and nav stay live. Progress/Week retire their bespoke inline `error` strings for the shared atom; Audit's read error is no longer swallowed (its "clean log" caught-up state stays a calm success, never confused with a failure); and a chapters-load failure on a book now surfaces on Book detail as a Tier-2 panel rather than an empty chapters list. A 401 from any of them routes to re-auth.
- **The error/notification model across the flat-list screens — Activities, Notes, Inbox, Connect, Queue.** Each now distinguishes **loading / empty / failed** via the shared `ui::panel` atom: a read that failed shows a loud red `✖ couldn't load …` + `r` retry instead of an empty list (Notes previously swallowed a failure into an empty list; Inbox/Connect/Queue's copy-pasted inline errors are unified). A 401 from any of them routes to re-auth (§SIGN IN · SERVER ERROR). Queue keeps its loud-vs-calm distinction (a corrupt queue is loud; a synced queue reads calm) now via the shared atom. Activities and Notes gain the search-state contract: the `/…▊` query caret in the title, matched-substring highlight, `n`/`N` stepping between matches, and a "no other matches" line.
- **The error/notification model, tier 3 — a blocking sign-in screen, and 401 → re-auth from anywhere.** New `ui::blocking` atom renders the design's §SIGN IN · SERVER ERROR: when the identity server is unreachable or returns 5xx, the sign-in flow shows a whole-screen failure (`✖ can't reach the identity server`) with a single `⏎ retry`, distinct from a cancelled/timed-out attempt (which returns to the idle prompt). And a **401 from any authenticated read now routes to re-auth** — `Action::SessionExpired` replaces the current screen with a "session expired — sign in again" prompt (`⏎` re-runs OAuth), instead of a confusing per-screen error. Books is the reference caller; other screens adopt the same 401 branch as they're touched.
- **The error/notification model, tier 2 — an inline "failed" state that never lies as "empty".** The design system (`design-system.dc.html` §ERROR & NOTIFICATION MODEL) splits a bordered region's no-rows body into distinct states: **loading**, **empty** (calm, muted), and **failed** (loud, red — the reason plus an `r` retry key). New shared atoms carry it: `ui::panel` (the Tier-2 body), `ui::search` (the `/…▊` query caret, substring highlight, and `n`/`N` match stepping), and `messages` — the one-spelling catalogue so the notify tile, the inline panel line, and headless `stderr` share the same words (design §C). **Books** is the reference adopter: a failed load now renders `✖ couldn't load books · identity… → HTTP 500 · r retry` instead of silently collapsing to an empty shelf, and `/rust` highlights matches while `n`/`N` step between them.

## [0.10.0] - 2026-07-17

### Added

- **The Queue inspector — the intent-log's own face, plus `engineer queue drop`.** EPIC #98's last surface: the offline write queue gets a TUI board (`:queue`, `g q`) and its headless verb set closes. The screen renders the **same `queue::pending()` read the CLI table prints** — the `#  INTENT  TARGET  AGE  STATE` columns are shaped once in a new `queue::view` module both faces call, so the board and `engineer queue` can never drift. Over that read, the design's §Queue inspector gestures: `j`/`k` move the full-row `▌` cursor; `r` **retry now** runs a reconnect drain through the shipped `drain_reporting` (the ambient replay transcript streams as intents land — the same tail the Timer reconcile runs); `x` **drop** the selected *diverged* write, armed then confirmed by a second `x`, routing the #109 `queue::drop_intent` (which refuses to orphan queued dependents); `⏎` on a diverged intent **opens the reconcile flow** — routed to the *shipped* reconcile panel on the Timer screen (#106/#109), never a second reconcile UI. Parked intents render dim with their reason (kept for review, out of the replay line); an empty queue reads calm (`queue empty · everything synced`); `q`/Esc close back to Home (the design's `q close` — the one screen where `q` isn't the global quit, since the inspector is a reached sub-surface). The headless twin gains **`engineer queue drop <id> [--force]`** — the standalone spelling of the board's `x`, a thinner shape over the *same* `drop_cmd`/`queue::drop_intent` as `resolve <id> --drop` (no logic duplicated, one confirmation contract). **Reachability decision:** the header's `↑N` / diverged chip stays a static status read — the TUI header isn't interactive, so there's no deep-link; the `g q` goto chord and the `:queue` palette verb (a full word with no `q`-alias, so `:q` still quits — exact beats prefix) are the way in. (`offline-write.dc.html` §Queue inspector; EPIC #98, #112 — the epic's final surface)
- **The long tail — one queue, not five.** Phase C of the offline-write epic: the remaining write paths adopt `QueuedClient`, closing the "same queue-and-reconcile shape for every mutation" promise. **Notes** — the browser's `a` archive/unarchive, the edit overlay's PATCH (including the #124 chapter/section anchor save), and `u` unlink all queue offline as `note:<id>`-stream intents with field-flip/echo synthesis; the `NoteInput` omit-vs-replace anchors contract rides the intent verbatim, so a queued edit that never touched anchors still leaves them untouched on replay. **Books** — a status flip (reading/done/hold) queues as `BookStatus` and replays plain (an upsert of a field, benign under re-fire). **Headless targets** — `engineer target declare/adjust/retire` now ride the same `QueuedClient` seam the Progress screen got in #121, so an offline declare queues with the `· queued (offline)` tail and `--json` gains the additive `queued` flag. **The last un-queued gestures** — the `a` capture form's create and the Activities table's `t` (start the timer on a row) — route through the seam too. **Recorded live-only, deliberately** (the can-we-synthesize-honestly test): a review *rating* (the server computes the next-due schedule a rating sets — an offline synthesis would lie about it), inbox triage accept/reject/ack (#94), capture-source connect/disconnect/sync (#95), note + segment hard deletes (#124's destructive-verbs rule), timer reclaim (the server's idle verdict, #104), and the presence heartbeat (meaningless offline). Each refuses offline with the way forward spelled out; the full sweep inventory — every mutation call site, queued or recorded — is on #111. (`offline-write.brief.md` §8 phase C; EPIC #98, #111)
- **The Phase B mop-up — `engineer plan add` and the activity lifecycle verbs go queue-aware.** The re-diff pass that closes the last un-queued Phase B write paths (much of #110's written scope had already shipped inside the surface epics: the Week board's plan declare/adjust/drop through `QueuedClient` #115, target writes #121, note capture #123, the week note #117, `engineer log` #108). Two gaps remained, now closed. **`engineer plan add`** routed its create straight through the plain API — it now rides `QueuedClient::create_activity` like the board's `a`, so an offline declare queues (a provisional negative-id row) and still exits 0 with the `· queued (offline)` tail; `--json` gains an additive `queued` flag (and the honest negative `id`) so a script can tell a deferred declare from a synced one. **The Activities table's row actions** — complete (`c`), archive/unarchive (`a`), duplicate (`d`) — called the plain API directly; they now route through `QueuedClient`, growing three additive, serde-tagged `IntentKind`s on the row's `"activity:<id>"` stream: `ActivityComplete`/`ActivityUnarchive` synthesize by field-flip (status→completed, archived→active) and replay **plain** — a second complete/unarchive is naturally idempotent server-side; `ActivityDuplicate` synthesizes a provisional negative-id copy and replays plain too. **The duplicate decision (recorded on #110):** duplicate is the one verb here that mints a *new* resource and is **not** in the server's `Idempotency-Key` opt-in set (ADR 0036), so a lost-ack re-fire on replay makes a *second* copy. That double-fire is an **accepted** risk over refusing the gesture offline — a duplicated activity is a visible, archivable planned copy (never double-*counted* like a logged segment), so the never-silently-lose-the-gesture invariant wins and the queue mechanics stay uniform across all four verbs. Provisional rows still refuse all four (they reference unminted ids — #109), so a queued lifecycle verb always carries a real id. (`offline-write.dc.html` §Log · provisional; EPIC #98, #110)
- **Provisional rows in reads — queued work is visible where you look.** The Activities table now folds the pending queue over every fetched page (`queue::fold_activities`, `fold_timer`'s sibling): a still-queued offline `ActivityCreate` renders as a full `◔ … provisional · queued` row mixed with the confirmed ones (amber mark, dim state, same table, one glyph — `offline-write.dc.html` §Segment audit · mixed), and a queued `SegmentCreate`'s minutes ride their parent row as `◔+Nm` beside the server-confirmed duration, never summed into it. Read-time composition only: nothing provisional is ever written into a cache, and the rows settle to plain on the very next fetch after the drain. Row actions (complete/archive/duplicate/timer) refuse on a provisional row — there is no server id to act on yet — and the detail opens from the local fold without a doomed refinement fetch. (EPIC #98, #109)
- **The rejected segment becomes a choice — edit / drop / skip (§Diverged · rejected segment).** A `422` on a replayed `SegmentCreate`/`ActivityCreate` (an overlap, a closed study day) now surfaces the design's case-B flow instead of a generic park-or-nothing. The reconcile panel grows a rejected-write face — the refused write's own identity (times, minutes, target), the server's objection verbatim, and three gestures: `e` **edit times** opens the payload's `started_at`/`minutes` (and a create's `title`/`planned_on`) in `$EDITOR` through the shipped hand-off plumbing, parses the saved buffer back under the writer lock, re-pends the intent with a **fresh idempotency key** (the corrected payload is a new logical write — the old key could replay the stored rejection or trip the server's key/body mismatch guard), and retries the drain; `x` **drop** is the queue's first genuinely user-chosen delete — armed then confirmed by a second `x` in the TUI, `--drop --force` headlessly, refused while queued dependents still reference the create's provisional id (dropping the parent would orphan them); `s` **skip & keep queued** parks it (reason `skipped · <objection>`), kept in `queue.json` and out of replay until a later decision. Headless twins on the shipped verb: `engineer queue resolve <id> --edit|--drop --force|--skip` (exit 4 when an edited retry diverges again). A parse-refused buffer or an aborted editor changes nothing — the intent stays diverged, loud, un-dropped.
- **`engineer log` works offline — provisional ids and replay id-mapping.** Phase B opens: after-the-fact capture survives a tunnel. Both `engineer log` shapes route through `QueuedClient` like every other write. `engineer log '<title>' --minutes N` stays a **single** create carrying `duration_minutes` (the server mints the completed activity *and* its opening segment from it — the composition that's actually there, not the design mock's literal two calls), so offline it queues one `ActivityCreate` and prints a provisional `· queued (offline)` line; `--json` carries the negative provisional id honestly (`queued: true`). `engineer log --activity '<fuzzy>' --minutes N` appends a segment through the new `QueuedClient::create_segment` / serde-tagged `IntentKind::SegmentCreate { activity_id, started_at, minutes }` on the `"activity:<id>"` stream; its fuzzy resolve is a **live** read, so offline it refuses with the way forward (`offline — can't resolve activity "…"; log a new one, or retry online`) rather than guess the wrong activity — the one spelling `timer start` and `note capture --book` already use. The segment replays under its queued `Idempotency-Key` (segment-create is in the server's opt-in set, ADR 0036), so a lost ack can never write it twice.
- **Per-stream FIFO — a divergence blocks its stream, not the queue.** #101's replay halted the whole drain at the first divergence; the pass now gates only the diverged intent's `stream`. The contract, documented on `queue::replay`: (a) in-stream order is never violated — nothing replays past the first diverged intent *in its stream*, in this pass or a later one; (b) streams are independent — a diverged `activity:<id>` write holds that activity while the `timer` stream, other activities, targets, notes, and week notes keep replaying; (c) the #108 id-map dependency edge counts as a stream dependency — an intent still referencing a provisional (negative) activity id is held explicitly (the parent create lives on the shared `activity` stream while the dependent keys `activity:-N`, so the negative reference *is* the edge; a blocked dependent never posts against a guessed id); (d) a transport failure still halts the whole pass — offline is global. `ReplayReport` stays honest: `diverged` is true while *any* stream waits on a choice, `replayed` counts across streams, so `✗ diverged — … 2 replayed · 1 still queued behind the choice` can now be literally true. `queue::fold_activities` mirrors the same ordering in reads: nothing folds past the first diverged intent in its stream, a dependent segment of a diverged create stays out of the picture, and parked intents never fold. (EPIC #98, #109)
- **The replay id-map — a queued segment finds its real parent.** A provisional resource now takes a **unique** negative id, `-(intent.id)` (not a flat `-1`), so a queued segment can reference a still-queued offline `ActivityCreate`. When that create replays and mints a real server id, the replay pass records `provisional → real` for the rest of the drain **and** stitches the real id onto every still-queued intent that referenced it — the segment, or an edit/archive of a provisional declare — **in the same writer-locked mutation that removes the acked create**. That one-lock write is the durability guarantee: a drain interrupted between the two writes leaves the queue already consistent, so a fresh drain (a new process, an empty in-memory map) replays the segment against the real activity id read purely from what was persisted. The map lives on the intents, never a second ledger — derived state stays derivable. Nothing double-writes on resume: the parent's key deduped it if it were re-sent, and the segment carries its own key. (`offline-write.dc.html` §Log · provisional; EPIC #98, #108)

### Added

Capture is **queue-aware** like every write: it routes through `QueuedClient::create_note` (a new additive, serde-tagged `IntentKind::NoteCreate { body }` on the `"note"` stream), so a loose thought captured offline queues and prints `· queued (offline)`, replaying when the wire returns. An anchored (`--book`) capture never reaches the offline arm — the book search that resolves the anchor is a live read, so offline it refuses with the way forward (`offline — can't resolve book "sicp"; capture loose or retry online`), exactly like a query'd `timer start`. The reads have **no offline story by design**: there is no note read-cache, so `list`/`search`/`show` refuse honestly offline (`offline — notes reads need the server`) rather than serve a stale answer. Archive-not-delete stays the vocabulary — this slice adds no destroy verb (the deliberate `delete`/`unlink`/anchor-picker faces are #124). One deliberate deviation from the design's §Headless mock: the ticket's `0`/`1` exit contract ships now; the design's richer script-facing codes (`2` empty result, `3` bad id, `4` editor aborted, `5` no editor) are deferred as a future refinement, recorded on #123. (`notes.dc.html` §Headless + §the one-line anchor read-back; EPIC #120, #123)

- **Target writes go queue-aware — declare, adjust & retire on Progress survive a tunnel.** The Progress screen's three target verbs (`n` declare, `e` adjust, `x` retire) now route through `QueuedClient` like every timer/week write: drain-before-live, enqueue on transport failure, loud if even queueing fails — an offline gesture never bounces. The queue's `IntentKind` gains additive, serde-tagged `target_create`/`target_adjust`/`target_retire` variants: a declare carries the whole `TargetCreate` body on the shared `"target"` stream and replays under its queued `Idempotency-Key` (keyed strictly dominates a plain re-send — where targets-create is in ADR 0036's opt-in set the server dedupes a lost ack, and where it is not the header is ignored; a replay re-sends the identical body under the identical key, so a key-reuse conflict cannot arise); adjust/retire key on their lineage row (`"target:<id>"`) and replay plain. Live, the same coded conflict renders the shared soft "moved on — re-fetching the live pace" copy and refetches, never a hard error. An offline declare renders a provisional `◔ scope · Nh/wk queued` line under the meters (the Week board's idiom) until it replays and a live refetch returns the real reading; offline adjust/retire confirm `· queued (offline) — will sync`. Retire stays retire — closes the lineage, keeps history, never deletes. (`progress.dc.html` §Declare panels; EPIC #119, #121)

- **The git-source connect flow — trust-gated, honest about what it can't do (`c` from the Inbox).** The epic's last surface, and the terminal-native half of assisted-capture: the step that makes drafts *appear*. A new capture-source client (`src/api/capture.rs`) over the shipped `/api/v1/capture/sources` contract (dsaenztagarro/engineer#804, engineer PR #808, ADR 0035) powers both a TUI screen and a headless twin. On the **Connect screen** (`src/app/screens/connect.rs`, reached from the Inbox via `c` — the design's footer key), a list of the capture sources (git / calendar) with their connect state, and three verbs behind a modal: `c` connects — rendering the source's plain-language **trust statement verbatim *before* connecting** (`reads` / `never_reads` / `promise`, straight from the payload — the CLI never invents its own wording), then a confirm (the git source takes no body; the calendar captures a feed URL first); `d` disconnects, behind a confirm that says what it means — **disconnect turns the source off, it doesn't delete the drafts it already captured** (disconnect ≠ delete); `s` enqueues a scan for a connected source. When the git source has no GitHub connection it is not `connectable`: `c` renders the server's **requirement pointer** honestly (the detail + the web URL — "connect GitHub on the web first") instead of offering a connect that would only `422` — GitHub OAuth is web-only (engineer ADR 0018) and the CLI never fakes a second auth path. The **headless twin** is the exact one-shot: `engineer inbox sources` (the sources with their state + trust `reads` line, or the requirement pointer), `engineer inbox connect <key> [--feed-url URL]` (prints the trust statement, then requires `--yes` to proceed non-interactively or confirms `[y/N]` on a TTY — the trust is flushed *before* the read, capture-is-sacred), `engineer inbox disconnect <key>`, and `engineer inbox sync <key>`; `--json` throughout, exit `0`/`1`, and the `422`s render their problem detail honestly (the git requirement, a bad calendar feed URL's field error, a sync-not-connected refusal). Like the triage verbs (#94), the connect verbs are deliberately **live-only** — not routed through `QueuedClient` — because an offline connect can't synthesize an opt-in that never happened: it refuses clearly (`offline — connecting needs the server; retry online`) and changes nothing. One shipped-time deviation from the §5-era mock: the design drew a per-repo picker against an unsettled contract; the shipped contract is a whole-source opt-in (no per-repo selection over the API), so the flow is a sources list with the trust gate, not a repo picker. (`assisted-capture.dc.html` §Connect · git source + §Headless; EPIC #118, #95 — the epic's final surface)

Needs the engineer API's new `POST /api/v1/timer/heartbeat` (dsaenztagarro/engineer#781), the last v1 gap versus the web timer — closing the follow-up left open when the idle guard shipped server-driven in #35.

### Changed

- **Design briefs reorganised — one brief per engineer-cli module, under a governing product principle.** The single omnibus `terminal-client.brief.md` — stale in both directions (the CLI had shipped most of its Phase-1 jobs; the `engineer` API had shipped `targets` / `today` / `weeks` / `automations` ahead of it) — is retired to a tombstone and decomposed into one brief per module: `timer` (absorbing the former `timer-gaps` gap analysis, its backend-API "Section C" now resolved), `progress`, `home`, `activities`, `review`, `notes`, `command-palette`, `week-planning`, `assisted-capture`, and a `cross-cutting` brief. Each is grounded in the API it actually consumes (verified against `engineer/config/routes.rb`) and states shipped reality rather than a pre-implementation snapshot; the pre-lifecycle `daily-loop.brief.md` is marked historical and the retired `books.html` / `terminal-tokens.css` anchors corrected to `design-system.dc.html` throughout.

- **A governing principle — "sterling, not a replica."** A new lead section in `cross-cutting.brief.md` (pointed to from the kit `README` and the briefs index) makes explicit that a full terminal replica of the `engineer` web UI is a non-goal: the terminal owns the study loop's high-frequency core as glances and gestures — the shipped timer is the exemplar — with depth staying on the web. Three surfaces were reframed to fit it: progress's time-explorer becomes a glance plus a `--json` rollup rather than a pivot grid, the activities table is capped as the one deliberate lean-back ledger (its watch-native core being capture and the audit), and week-planning is a planned-vs-done readout plus a one-liner declare rather than a planning canvas.

## [0.3.0] - 2026-07-06

### Added

- **The focus session has a face.** When the live timer is in focus mode (today: started from the web; the CLI start joins once the focus API ships), the watch face reads it faithfully: `◆ WORK · INTERVAL n` with the accent digits while counting, and a muted `○ BREAK — NOT COUNTING` face with the reminder that a break is never a segment. The instrument rail gains the POMODORO block — banked intervals as green dots, the live one accent (hollow on a break), no invented remainder since the round length is a settings knob the API doesn't serve yet. `n` (skip interval) joins `m` in naming the missing focus API instead of silently doing nothing; the header cell's `◆`/`○ break` forms light up from the same read. (`timer.dc.html` §Focus rhythm)

- **Bind-at-stop, undo, and the discard that asks twice.** Ending an *unnamed* timer no longer dead-ends in a warning: `s` freezes the clock (a real pause, so deciding time never pollutes the segment) and opens the bind picker — `⏎` binds and saves in one stroke, `Esc` resumes exactly what the panel froze (a clock you'd paused yourself stays paused). The stop confirmation now carries **undo**: `u` deletes the just-written segment — the exact inverse of the save — and returns to the empty face. And `d` discard works on any live timer but respects real work: past ~2 minutes the first press warns (`discard 41m of work? d again to confirm`), only the very next `d` confirms, and any other key disarms it — the same fence `engineer timer discard --force` mirrors headless, now shared as one constant. (`timer.dc.html` §Bind at stop / §Saved & undo)

- **The start picker — one list, every way in.** `s` on the empty timer screen (or `/` from the live face) opens the §Start-a-timer picker: a live fuzzy search over your bindable activities, a `＋ new activity` row that creates & starts from the typed title, and a `▶ just start` row for an unnamed clock you bind when you stop. `Tab` toggles Stopwatch ⇄ Focus for the fresh start — picking Focus currently explains that focus-start needs the focus API rather than pretending. With a timer already running the picker becomes the stop-&-switch flow: the list narrows to existing activities, the first `⏎` shows the "One timer at a time" banner naming what's running and what you picked, the second `⏎` stops & saves the current timer and starts the new one (`Esc` steps back without touching the clock) — the same `switch` semantics as `engineer timer start --switch`. (`timer.dc.html` §Start a timer / §Start conflict)

- **The timer status line speaks every state.** The persistent header cell (every screen, right-aligned) now renders the full grammar from the timer redesign: `●` running in green with the muted activity title (truncated past 24 chars), italic *untitled* for an unbound clock, `‖` paused with the frozen clock muted, `◐` idle with an amber ` idle ` pill when the server flags a reclaim pending, `◆` focus work with pomodoro dots (banked green, live accent), and `○ break … not counting` muted. Below ~70 columns the cell drops to the narrow glyph-plus-clock form so the breadcrumb keeps its room — the fixed-footprint promise. This supersedes the v1 pill's "never a title" rule by design (`timer.dc.html` §Status line); the `over` form lands with the overrun feature.

- **The timer screen is a watch face.** The live timer renders as the redesign's hero (`timer.dc.html` §Timer hero / §Paused): one big block-digit number (accent while counting), a state label, the "since HH:MM" context, and the bound activity — or italic *untitled* with the bind hint. `i` folds the instrument rail (a TODAY total summed from today's activities; the focus instruments arrive with the focus display) and the number recenters into the calm face. `SPC` is pause ⇄ resume on the live face (`p` stays as an alias), and pausing freezes the whole face amber — `‖ PAUSED — NOT COUNTING`, muted digits, and the reminder that a paused timer never goes idle. `s` is the stage-dependent primary: start the clock when absent, end & save when bound, and the bind-first warning when unbound (`x` remains a hidden legacy alias). `m` explains that mode switch needs the focus API rather than silently doing nothing.

- **Headless `engineer timer` — every timer verb has a one-shot twin.** The live timer is now fully drivable from scripts and status bars, no TUI required: `engineer timer` prints a one-line human read (`--json` for the full read: state word, mode/phase, intervals, activity, timestamps, elapsed); `engineer timer status` emits the fixed-order contract line `<state> <elapsed_s> <mode> <activity_id> <kind> "<title>"` (with `-` placeholders when unbound) and `--short` the glyph-plus-clock status-bar form (`● 52:14`, empty when nothing runs) — the same glyphs as the in-app header cell. Write verbs: `start [query]` fuzzy-binds via `/timer/candidates` (bare `start` begins an unnamed timer; a conflict refuses and names what's running, `--switch` stops & saves it first), `toggle` (pause ⇄ resume, made for tmux/zellij keybinds), `pause`, `resume`, `stop` (refuses on an unbound timer with "bind or discard first"), `bind <query>`, and `discard` (past ~2 minutes it refuses without `--force`, the headless form of the TUI's confirm). Exit codes answer *is the clock counting?* — `0` counting (running/focus work) · `1` nothing running or a refused write · `3` idle, reclaim pending · `4` paused or on a break — and output is plain when piped (ANSI only on a TTY, `NO_COLOR` honoured). (`timer.dc.html` §Headless / §Headless contract)

- **Book status picker modal — the last of the daily-loop screens.** Pressing `s` on a book's detail opens a small centered modal listing the five statuses as their kit pills (` reading ` accent · ` done ` success · ` unread ` muted · ` hold ` warn · ` stop ` danger), opened at the book's current status. `j`/`k` step the highlight and the `r`/`c`/`u`/`h`/`a` mnemonics jump straight to a status; `↵` confirms and `Esc` cancels. Confirm PATCHes `/api/v1/books/{id}` via the existing `update_book`, the detail header reflects the new status, and a notify tile confirms (`status → on hold`). While the modal is open it owns those keys (so `h` picks `on_hold` rather than stepping back, and `Esc` closes the modal, not the screen). This finishes the sketched-but-unbuilt TODO on the book detail screen and completes EPIC #7's phased plan for the terminal daily loop.

- **The `:` command palette, completed as one grammar table.** The command line now knows the whole daily-loop vocabulary pinned by the brief (§5, `command-palette.html`), and the whole thing is driven by a single source of truth — `src/app/command.rs`'s `ENTRIES` table (verb → kind → argument shape → target + one-line help). The dispatcher, Tab completion, the inline hint, and `:help` all read that one table, so what runs can never drift from what the UI advertises (the old scattered `if`-chain in the submit handler is gone). **Navigation** covers every screen — `:home` `:books` `:activities` `:notes` `:review` `:progress` `:timer` (bare verb goes to the screen); **actions** are `:timer start|pause|resume|stop` and `:note <text>`; **housekeeping** is `:q`/`:quit` `:logs` `:w` `:logout`, plus a new `:help`. Timer actions dispatch against the app-owned timer snapshot from *any* screen (no navigation, no screen-load race) — the header cell shows the result, and an invalid transition surfaces the same notify-tile warning the Timer screen gives (e.g. stopping an unbound clock says "bind the timer before stopping"). `:note <text>` opens the quick-capture overlay **prefilled** with the text (a *new* draft, not a direct create) so a thought can grow an anchor or be saved with one keystroke. The line reads its four states from the table: an empty `:` advertises the top verbs, a partial verb shows the matches and completes on **Tab** to their longest common prefix (vim's `wildmode=longest`; stateless, no cycle to track), an unknown verb is helpful not hostile ("unknown :xyz — try :help"), and executing shows the normal result. **Abbreviations** resolve as in vim: an unambiguous prefix wins (`:act` → activities, `:t start` → timer start, `:timer p` → pause), while an ambiguous one lists its candidates instead of guessing (`:l` → logs/logout, `:h` → home/help). The unpinned `:activity`/`:a` verbs were dropped (the +activity form stays on the `a` key), which is what lets `:act` and `:a` resolve to activities.

- **Review screens — dashboard, the rating sitting, and browse (`R` from Home, the `<Space>` leader `R`, or `:review`).** The Review pillar's first UI: one screen, three stages. The **dashboard** is a minimal read — the due-queue count and estimated minutes as the call to action, the streak stats (`streak N days · best M · X this month · avg Yd interval`), and the due queue in urgency order as a preview. No ASCII heatmap: the streak and this-month counts already convey cadence in the terminal, and the web app owns the heatmap (the `dashboard.heatmap` payload is parsed but deliberately not rendered). The **sitting** is the payoff: `↵` (or `s`) starts at the queue head, each topic shows its name, domain, and last-reviewed/interval context (plus prompts and the per-rating interval forecast when the payload carries them), and you rate with a single keystroke — `f` forgot · `z` fuzzy · `s` solid · `i` instant (chosen so the ratings never collide with the global `r` refresh). The server returns the next due topic and the sitting advances automatically until the queue drains to a quiet "done" view (rated count); `Esc` exits mid-sitting cleanly, since each rating is already committed per topic. **Browse** is the secondary state (`b` from the dashboard): a paginated topic table where `s` cycles the API's sort ring (urgency → recent → most/least-reviewed → longest-interval → a–z), `/` maps to the server-side `q` search, `]`/`[` step pages (the Activities/Progress precedent), and `↵` opens a topic detail read with a one-off rate option. The API client (`src/api/review.rs`) was vendored byte-identical from the user's WIP tree (the #9/#10 discipline), diff-verified before and after the work.

- **Activities table screen (`A` from Home, or `:activities`).** The core domain surface: a dense, scannable ledger of recent activities — a semantic status pill, kind, title, domain (by name; the terminal palette has no per-domain colours), duration, and a relative "when" (`5m ago`, `2d ago`, then a date). It is the **first surface to expose `meta.page`**: the panel's bottom border reads `page N of M · X total`, and `]`/`[` step pages (consistent with the Progress screen's week nav). `f` cycles a single TUI-light filter ring (all → planned → started → completed → archived) that folds the server's status and archived axes into one key; `/` filters the loaded page client-side across title, kind, and domain (the list API has no text search). Row actions: `c` completes, `a` archives/unarchives (reversible, so it toggles quietly with no confirm), `d` duplicates ("do this again"), `↵` opens a full-field detail read (kind/intent/bloom, domain, duration, segment count, timing, generated notes), and `t` starts the live timer bound to the selected activity (the header cell refreshes). Mutations refetch the current page rather than patching a row, so the ledger always mirrors the server. `A` opens the table while `a` stays "+activity" (the new-activity form), keeping muscle memory intact. Home's collapse to a single `GET /api/v1/today` call is deferred until that endpoint ships (engineer#663).

- **Quick-capture overlay + notes browser.** Capturing a note now costs a couple of keystrokes from anywhere: `<Space>c` (or `c` on Home/Notes) opens a modal quick-capture overlay over the current screen — a multiline thought, an optional book anchor (a live search over your books, the timer bind-panel idiom), and an optional page. Save is explicit (`Ctrl-S`, since Enter is a newline in the editor); **capture is sacred** — an accidental `Esc` on a non-empty draft only warns (`Esc` again to discard), so input is never lost. On save the note POSTs to `/api/v1/notes`, adding a book + page anchor when both are set. A new **Notes browser** (`n` from Home or the `<Space>` leader, or `:notes`) lists notes with a one-line anchor read-back for anchored notes (`SICP · ch 3 · p.142`) next to loose thoughts, `/` searches the server, `j`/`k` navigates, `↵` opens a full-content detail read, `a` archives/unarchives, `t` reveals archived notes (dimmed in place), and `e` reopens the same overlay pre-filled to edit (PATCH).

- **Timer screen + persistent header cell.** A new `Timer` screen owns the daily-loop timekeeping moments: start a blank clock ("name it later"), bind a running timer to work by searching `/api/v1/timer/candidates` or minting a new activity from a typed title, pause/resume, and stop — which shows the written segment (minutes → activity) so the ledger is trusted. On top, a fixed-width **header cell on every screen** renders the live timer per the web pill contract: an accent `● mm:ss` (widening to `h:mm:ss`) when running, an amber `‖ mm:ss` when paused, and nothing when absent — never the activity title. The snapshot lives in app state and is polled every 15s from `GET /api/v1/timer`, with the displayed elapsed ticked locally between polls. Reachable from Home with `t` (or the `<Space>` leader `t`, or `:timer`); on-screen keys are `s` start · `/` bind · `p` pause/resume · `x` stop · `d` discard.

- **Progress screen (`engineer pace` meters).** A new read-only screen renders the weekly pace aggregate from `GET /api/v1/progress` as one ASCII meter per target, behind-first: a block-bar with a now-tick (where the week expects you to be), `actual/target` hours, the signed delta, and the pace state (`behind` amber / `on pace` green / `met`). A week-header line (`2026-W27 · sat · day N of 7 · now = NN%`), a behind-total footer naming the largest gap, and a compact kind-mix line round it out. Reachable from Home with `p` (or the `<Space>` leader `3`/`p`); step weeks with `[` / `]` and return to the current week with `t`. Pace is derived from segments at read time, so nothing is stored client-side.
